# ArangoDB REST API Endpoint Permissions

This document maps every REST API endpoint to its required authentication and
authorization checks. It was generated by analyzing the handler registrations
in `GeneralServerFeature.cpp` (and engine-specific files) and then inspecting
each `RestHandler` subclass for calls to `ExecContext` methods.

**Legend for permission levels:**

| Symbol | Meaning |
|--------|---------|
| **NONE** | No authentication required (path bypasses auth layer) |
| **ANY** | Any authenticated user (no further in-handler checks) |
| **DB:RO** | `canUseDatabase(auth::Level::RO)` |
| **DB:RW** | `canUseDatabase(auth::Level::RW)` |
| **COLL:RO** | `canUseCollection(name, auth::Level::RO)` |
| **COLL:RW** | `canUseCollection(name, auth::Level::RW)` |
| **ADMIN** | `isAdminUser()` — user with write access to `_system` database |
| **SUPERUSER** | `isSuperuser()` — authenticated with JWT token, empty `preferred_username` |
| **HARDENED** | `canAccessHardenedApi()` — open normally, ADMIN when `--server.harden` is on |
| **_system** | Endpoint requires being called in the `_system` database context |

## 1. Auth Bypass Paths

Before any handler runs, `CommTask::canAccessPath()` (`arangod/GeneralServer/CommTask.cpp:786-876`)
decides whether the request needs authentication. The following paths **bypass authentication entirely**:

| Path | Behavior |
|------|----------|
| `/` | Root path — granted superuser |
| `/_open/*` | All paths under `/_open/` — granted superuser |
| `/_admin/aardvark/*` | Web UI static assets — granted superuser |
| `/_admin/server/availability` | Health-check endpoint — granted superuser |
| Non-`/_` paths (Foxx apps) | Only when `--server.authentication-system-only true` — granted superuser |

Additionally, `POST /_api/user/<username>` is partially accessible without auth (grants read-only context for password verification).

## Appendix: Notable Alternative/Combined Permission Patterns

1. **`/_admin/cluster/moveShard`** — **ADMIN** OR **COLL:RW** on the target collection. This is an **alternative** auth pattern: you need either admin rights OR write access to the specific collection being moved.

2. **`/_api/replication/restore-*`** — **DB:RW** OR **COLL:RW** depending on overwrite mode + **ADMIN** escalation. Admin users are automatically elevated to superuser scope.

3. **`/_api/dump/start`** — **COLL:RO** per collection being dumped + **ADMIN** escalation on single servers. Each collection is checked individually.

4. **`/_api/query-plan-cache` GET** — **DB:RO** + per-collection **COLL:RO** filter. Superusers see all plans; other users only see plans for collections they can read.

5. **`/_api/view` GET (list)** — **DB:RO** + per-view check. Views backed by collections require the user to have at least COLL:RO on the underlying collections.

6. **`/_api/collection` GET (list)** — Each collection individually filtered by **COLL:RO**. Users only see collections they have access to.

## Migration to RBAC

We should map the permissions in the following way to RBAC actions and resources:

 - `NONE` `-->` stays as it is
 
 - `ANY` `-->` stays as it is
 
 - `DB:RO` `-->` action `db:ReadDatabase` with resource `db:database:<dbname>`
  
   whatever needed this will generally still need it.
   
 - `DB:RW` `-->` stays as it is
   
   whatever needed this will generally still need it.
 
 - `COLL:RO` `-->` split into `db:ReadCollection` (meta data) and `db:ReadDocuments` (documents)
   
   for whatever needed `COLL:RO` we should decide if it needs data or meta and change
   accordingly, without RBAC, both map to the old collection read access.
 
 - `COLL:RW` `-->` split into `db:WriteCollection` (meta data) and `db:WriteDocuments` (documents)
 
   for whatever needed `COLL:RW` we should decide if it needs data or meta and change
   accordingly, without RBAC, both map to the old collection write access.
 
 - `ADMIN` `-->` generally mapped to a new action `db:admin-*`, mostly without a resource,
   we might want to group certain APIs into one action as for example `db:admin-monitoring`,
 
   `==>` This area will need a lot of attention and specification, which I will work on next.
 
 - `HARDENED `-->`

## 9. Complete Endpoint–Action Reference Table

Ideas:

 - `NONE` stays no authentication required
 - `ANY` stays any authenticated user (no further checks in handler)
 - `DB` access stays as it is
 - `COLL` access distinguishes meta data and document data
 - views and analyzers get their own actions like collection meta data
 - `ADMIN` is split into many actions with prefix `db:Admin`
 - `HARDENED` is considered to be always on for RBAC, so becomes the same as `ADMIN`
 - `SUPERUSER` stays superuser only

Note that some APIs are marked "ANY", but this is **not** dangerous because
  - the API is only available in MAINTAINER_MODE
  - the API is only available on DBServers or agents, which do not have users, so they
    only accept SUPERUSER anyway
  - the API is only compiled in when FAILURE_TESTS are activated

Others are marked `ADMIN` but run only on DBServers or agents, so that they actually only
accept SUPERUSER anyway.
  
Furthermore, there are some command line switches, which change authentication behaviour,
in a lot of cases these have 3 possible values: `SUPERUSER`, `ADMIN`, `ANY`, sometimes
it is possible to switch off the API entirely. These switches remain and take precedence.
RBAC will only be considered if the switch is on `ADMIN`.
